Trestle — Feature Architecture & Scaling Roadmap

Planning model structure, admin capabilities, and path to enterprise scale (100-1000 users)

1. Planning Model Structure

Workspace Architecture

One shared workspace per year containing 14 plan files:
1 × AOP 12 × RF (monthly) 1 × LTP
Each plan file holds a complete state_data JSON with all input modules. Scenario types: budget forecast actual ltp

AOP — Annual Operating Plan

Full-year budget across all accounts
Exec Summary Budget
Modules: Revenue, HC, Vendor, CTR, T&E, Other, D&A

RF — Rolling Forecast

Monthly re-forecast with actuals split
Exec Summary N+M per month
Labels: 1+11, 2+10, 3+9... (actuals+forecast)

LTP — Long-Term Plan

Multi-year forecast (5 years)
Exec Summary Forecast
Growth assumptions, Y/Y comparisons

4. Current Admin & Access Controls

What Exists Today

Admin code login (hardcoded)
Module access toggles per user email
Ops view restriction per user
Plan sharing (owner/editor roles)
WebSocket presence + live cursors
Functional pillar assignments

Current Limitations

No row-level permissions
Single JSON blob per plan (no partial saves)
No audit trail in DB (only in state)
No SSO / OAuth (email-only auth)
No rate limiting or session tokens
No multi-region / read replicas

5. Scaling Roadmap: 10 → 100 → 1000 Users

Phase 1 — Foundation (10-50 users)

Authentication & Sessions

OAuth (Google/Azure AD) replacing email-only login
JWT tokens with refresh rotation
Server-side sessions (express-session + Redis)

Data Model

Normalize state_data into relational tables
employees, vendor_rows, plan_config as separate tables
Row-level saves (PATCH instead of full PUT)
Phase 2 — Enterprise (50-200 users)

Access Control

Row-Level Security (PostgreSQL RLS policies)
Dimension-based access: user → market/bizline/project
Module-level read/write/hidden per role
Admin panel with user management UI

Collaboration

Conflict resolution (OT or CRDT for cell-level edits)
Edit locking at row/section level
Audit log persisted to DB with user + timestamp
Phase 3 — Scale (200-1000 users)

Infrastructure

Redis for caching + session store
Read replicas for reporting queries
WebSocket clustering (Redis pub/sub adapter)
Rate limiting + request queuing

Integrations

REST API for EPM push/pull
Actuals ingestion pipeline
Export to EPM (Oracle, Anaplan, Workday Adaptive)
SSO + SCIM user provisioning
