Trestle — Executive Brief

For finance and finance-tech leads. Frames what the platform does today, where it's going, and the financial structure underpinning both. Plain language; technical terms defined in footnotes where unavoidable.

1. What Trestle Is

Trestle is a planning platform for finance teams. Each fiscal year lives in one shared workspace that holds 14 plans: one Annual Operating Plan (AOP), twelve monthly Rolling Forecasts (RF), and one Long-Term Plan (LTP). Every plan is built from the same set of input modules — revenue, headcount and compensation, vendor spend, contractors, travel and expense, other costs, and depreciation — coded against a common chartfield grid so any view rolls up consistently.

Output is the same for every plan: a P&L walk, a pivot/analytics layer for slicing the numbers, and scenario comparisons between budget, forecast, and actuals. Multiple users edit the same plan at the same time with live presence indicators.

2. Who Uses It

FP&A planners

Build the AOP, refresh the monthly Rolling Forecast, run scenario compares, and produce P&L variance views for leadership.

Finance partners (BU / function)

Maintain the parts of the plan they own — headcount for their team, vendor spend for their cost centers — with chartfield coding handled in-line.

Finance leadership

Reviews scenario comparisons, P&L walks, and FTE / cost rollups across business units, markets, projects, and accounts.

Finance-tech / admin

Maintains shared reference data (rate cards, benefits load, salary bands, vendor catalog), publishes new plan templates, and manages user access.

Chartfield grid (the coding spine)

Every line in every module is coded against the same five dimensions. This is what allows any number to roll up cleanly into the P&L and any pivot view.

Business Unit Business Line Market Project Account

A single rate card change, salary band update, or vendor consolidation flows through every plan that uses those dimensions — no parallel updates required.

4. Granularity — What Sits Behind Each Line

Trestle plans at the line level, not the rollup level. That means each cost line in the P&L is built from real underlying records — the planner can drill from any total back to the contributing rows without leaving the platform.

ModuleWhat "one row" represents
RevenueOne revenue stream per chartfield combination, with monthly amounts
HeadcountOne employee — name, salary, function, country, capitalization %, allocation across pillars
VendorOne vendor relationship per parent company per chartfield
ContractorOne contractor with rate, hours, start / end month, capitalization split
T&E / OtherOne expense line with type, description, chartfields
D&A / AssetsOne asset with depreciation method, cost, useful life, in-service date

This is the difference between Trestle and a top-down spreadsheet: every line is auditable to its underlying drivers, and changes in one driver (e.g., a salary band update) propagate consistently.

5. Features (User-Facing Capabilities)

Spreadsheet-style entry

Every grid behaves like a spreadsheet — keyboard navigation, range selection, copy/paste, bulk delete — so finance users move through the plan at the speed they expect.

Real-time multi-user editing

Multiple users edit the same plan simultaneously. Live cursors show who is in which section. State stays in sync without manual refresh.

Scenario comparison

Compare any two plans (Budget vs Forecast, Forecast vs Actual, Forecast vs Forecast) with variance overlay on every view.

Pivot and analytics

Reshape any module's data by Product, Function, Business Line, Market, Pillar, Month, or Quarter. Charts include bar, line, bubble, and scatter; calculated metrics support custom ratios.

P&L walk

Every input module rolls into the P&L automatically — no separate consolidation step. Subtotals (gross margin, opex, EBITDA) are produced from a single source of truth.

Export

Any view exports cleanly to Excel or PDF; charts copy to clipboard for quick reuse in board materials.

6. Current State vs Ideal State

A side-by-side of what exists today and what the rebuild is targeting. Same conceptual model — the change is in robustness, scale, and integration, not in how a planner thinks about a plan.

Today

Plan structure

  • 14 plans per year (AOP, 12 RFs, LTP) in one shared workspace
  • All input modules and chartfields wired in

Collaboration

  • Live presence and cursors
  • Last edit wins under simultaneous saves1

Access control

  • Single admin login + email-based access toggles
  • Plans shared with viewer / editor roles

Reference data

  • Salary bands, rate cards, benefits exist informally — not centrally maintained

Integration with other systems

  • Manual import / export to ERP, HR, and EPM systems2

Audit

  • Limited — change history is in-session, not persisted

Ideal state

Plan structure

  • Same 14-plan model — no change in the planner's mental model
  • Each module saved as discrete records, not as one big plan blob — enables row-level undo, partial saves, and faster loads

Collaboration

  • True cell-by-cell merge — no lost edits when two users save at once
  • Optional row / section locking during close periods

Access control

  • Single sign-on (SSO)3 with corporate identity
  • Access scoped by chartfield dimension (e.g., a planner sees only their market or business line)

Reference data

  • Centrally maintained, versioned reference sets — rate cards, benefits load, salary bands, FX rates, vendor catalog, GL chart of accounts
  • Plans pin to a version so a published rate change does not retroactively alter a closed plan

Integration with other systems

  • Standard, versioned connectors to ERP, HR, EPM, BI, and data lake
  • New firm onboarding = configure connector, not custom build

Audit

  • Full change history persisted — who changed what, when, and from what value

7. Roadmap — Pilot to ~40 Analyst Users by EOY

Current scope is one midsize organization. The pilot runs on a single team within that org; full rollout reaches roughly 40 analyst users by year-end. A broader opportunity (a few hundred or more users across adjacent teams or the wider firm) sits beyond the current engagement — design and infrastructure choices are sized to make that expansion possible without rework.

Phase 1 — Pilot (1 team, in-org)

Goal: get the full target stack working end-to-end on one team, with everything below in place before opening up to the rest of the org.

  • SSO login with corporate identity
  • Module-level and dimension-level access controls
  • Cell-level real-time merge — no lost edits under concurrent editing
  • Persisted change history
  • Centralized reference data (rate cards, benefits, salary bands, FX, vendor catalog)
  • At least one inbound and one outbound integration validated end-to-end
  • Automated test suite covering calc accuracy, save reliability, and integration round-trips

Exit criteria: a full close cycle run on Trestle with zero data loss, validated integration round-trips, and a green automated regression suite.

Phase 2 — Org rollout to ~40 analyst users by EOY

Open the platform to the rest of the organization in measured cohorts after the pilot exit criteria are met. Each cohort has explicit checks before the next opens.

CheckpointWhat gets validated before adding more users
Pilot → first additional teamReference data published org-wide · access scoping verified · save / merge stability under double the pilot load
~20 usersConcurrent-edit stress · integration freshness checks · audit log completeness
~40 users (EOY target)Capacity headroom plan · documented runbooks · close-cycle dry run with the full org cohort

Beyond current scope

A few hundred or more users in adjacent teams or across the broader firm is a real opportunity but is not in current scope. Design and infrastructure decisions taken in Phase 1 (normalized data, dimension-scoped access, versioned connectors) are deliberately sized to support that expansion — so the path is shortened, not pre-built.

8. Expected Time Savings

Roughly 100 hours recovered per analyst, annually, once Phase 1 capabilities are live across the org. Estimate based on observed pilot workflows.

  • ~60% of the savings sit in the AOP cycle — planning entry, reconciliation, scenario compare, and review cycles compress meaningfully. The remaining ~40% is spread across monthly Rolling Forecast and Long-Term Plan work.
  • ~70% of the savings are driven by streamlined load and system integration tools — the spreadsheet-style grid plus standard connectors to ERP, HR, and EPM systems remove most of the manual import / export and reconciliation work that dominates the current cycle. The remaining ~30% comes from real-time collaboration, scenario tooling, and reduced rework from concurrent-edit issues.

The two cuts are different views of the same 100 hours — by planning phase (AOP vs RF vs LTP) and by feature category (load & integration vs other) — so they overlap rather than add. The single biggest block is AOP work that gets faster because integration tools eliminate manual reconciliation against ERP and HR data.

At the EOY scope (~40 analysts), this scales to roughly 4,000 hours per year recovered across the organization.

9. What This Buys the Business

For finance teams

  • Faster close — fewer manual reconciliations, fewer parallel spreadsheets
  • One source of truth for plan, forecast, and actuals
  • Drill-through from any P&L line to the underlying drivers in seconds
  • No more lost edits during concurrent close work

For the firm

  • Consistent chartfield discipline across business units
  • Audit trail that satisfies internal review and external audit requests
  • Standardized integration with the firm's systems of record — no bespoke pipelines per team
  • Rate cards, benefits, and FX maintained once and used everywhere

Footnotes

1 Last edit wins. When two users save changes to the same plan at the same time, only the most recent save is kept; the other user's changes are silently dropped. The rebuild replaces this with cell-level merging.

2 ERP / HR / EPM. ERP = the firm's general ledger system (e.g., SAP, Oracle, NetSuite). HR = the workforce system of record (e.g., Workday). EPM = enterprise performance management — consolidation and reporting tools (e.g., Oracle EPM, Anaplan, Workday Adaptive).

3 Single sign-on (SSO). Users log in once with their corporate credentials (e.g., Google Workspace, Azure AD) and access Trestle without a separate password.
