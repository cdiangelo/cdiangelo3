Trestle — Technical Brief

Companion to the executive brief. Adds data shapes, calc rules, architecture details, and a submodule inventory under the new gauge framework (complexity 1-5 + "similar to" tags). Ends with deep dives on the five priority submodules.

1. System Overview

Multi-user FP&A planning platform. One shared workspace per fiscal year holds 14 plan files: 1 AOP + 12 monthly RFs + 1 LTP. Each plan file is a state_data JSON containing every input module (revenue, HC, vendor, contractor, T&E, other, D&A) coded against a five-dimension chartfield grid (BU / Biz Line / Market / Project / Account).

Calc outputs feed a P&L walk and a pivot/analytics layer. Real-time collaboration runs over a WebSocket connection with presence and live cursors; persistence today is full-state PUT per plan, with row-level PATCH and per-cell merge as rebuild targets.

2. System Spine

Plan container

Workspace (year) holds 14 plans. Each plan is one state_data blob. Scenario types: budget forecast actual ltp. RF labels follow N+M convention (e.g., 3+9 = 3 actuals + 9 forecast months).

Input modules

Revenue · HC & Comp · Vendor / OAO · Contractors · T&E · Other · D&A. Each row coded to chartfields plus jan-dec month columns.

Outputs & collab

P&L walk · pivot & analytics · scenario compare. Multi-user via WebSocket presence + live cursors; full-state broadcast today, per-cell CRDT target.

4. Current vs Target Architecture

Today

  • Persistence: single state_data JSON blob per plan, full PUT on save
  • Auth: hardcoded admin login + email-based module toggles
  • Sync: WebSocket full-state broadcast; last-write-wins under concurrent saves
  • Audit: in-memory only, not persisted to DB
  • Reference data: embedded constants and informal lookups
  • Integrations: ad-hoc per integration; not versioned
  • Reporting: client-memory aggregation from state_data

Target

  • Persistence: normalized tables (employees, vendor_rows, plan_config, field_defs) with row-level PATCH and version columns
  • Auth: OAuth (Google / Azure AD), SSO + SCIM, JWT with refresh rotation, server-side sessions in Redis
  • Sync: per-cell CRDT doc per plan, Redis pub/sub fan-out, optimistic locking
  • Audit: append-only change_log partitioned by month (user, field, before, after, ts)
  • Reference data: versioned reference_sets; plans pin to a version
  • Integrations: versioned connector modules with schema map, chartfield crosswalk, retry / idempotency, reconciliation report
  • Reporting: server-side aggregation, materialized views per scenario, read replicas for reporting
  • Access: PostgreSQL Row-Level Security (RLS) policies enforce dimension scoping

6. Deep Dives

First-pass depth on the five priority submodules. Intentionally shallow — we iterate from here.

6.1 HC & Compensation

Data shape

employees[] — name, salary, seniority, function, country, capPct, allocations[], start mo, end mo, bonus %, equity %.

Calc rules

  • Monthly proration based on start / end month (mid-month treated as full month today; flag for revisit).
  • Bonus % posted as annual amount to a configurable target month.
  • Benefits load applied as country-specific % on top of base salary.
  • CapEx % (capPct) splits cost into CapEx and OpEx; flows into D&A pipeline.
  • Allocation % across functional pillars routes cost into pivot dimensions.

Dependencies

Salary bands · benefits load by country · FX rates (target) · pillar map · GL accounts.

Today vs target

  • Today: nested in state_data; full-plan blob save on every edit.
  • Target: dedicated employees table with row-level versioning; reference_sets for bands / benefits / FX; per-employee audit trail.

Edge cases & opens

  • Mid-month start / end and partial-month proration policy.
  • BU / market transfers mid-year (split row vs allocation change).
  • Equity refresh accounting and overlay vs base.
  • Contractor → employee conversion and continuity.
  • Multi-currency comp and FX restatement.

6.2 Collaboration & Presence

Today's shape

  • WebSocket presence + live cursors.
  • Full state_data JSON saved on change; last-write-wins.
  • No cell-level merge; no row-level locking.
  • Audit trail in client state only.

Problems

  • Lost edits under concurrent saves.
  • Large WS payloads as plans grow.
  • No persistent audit.

Target architecture

  • Per-cell CRDT (or OT) doc per plan; server-authoritative merge.
  • Redis pub / sub fan-out across app instances.
  • Optimistic locking with version column.
  • Append-only change_log partitioned by month.

Flow

Client emits ops → server validates against tenant + RLS → applies to CRDT doc → broadcasts via Redis → other clients merge → batched persist.

Risks & opens

  • Replay correctness across schema evolution mid-edit.
  • Presence performance at hundreds of users.
  • Audit log growth and partitioning policy.
  • Open: which CRDT lib · cell vs row granularity · "X is editing Y" UX detail.

6.3 P&L Walk

Inputs

Every module's calc output: Revenue, C&B (from HC), OAO (from Vendor + Other), CTR (from Contractors), T&E, Other C&B, D&A.

Output

Chartfield × month grid with subtotal hierarchy (gross margin, opex, EBITDA) and scenario variance overlay.

Logic

  • Sum by chartfield × month per line.
  • Subtotal hierarchy defined per tenant template.
  • Loaded-cost rollups (e.g., bonus + benefits into C&B).
  • Scenario compare (budget vs forecast vs actual) with variance overlay.

Today vs target

  • Today: computed in client memory from state_data.
  • Target: server-side aggregation against normalized tables · materialized views per scenario · cached at chartfield rollup level.

Risks & opens

  • Any module shape change ripples here — needs contract tests against golden P&L fixtures.
  • Open: report builder vs hard-coded P&L lines · tenant-specific subtotal hierarchies · drill-through UX.

6.4 EPM & Integrations

Inbound

  • ERP actuals (SAP / Oracle / NetSuite) → state.actuals
  • HR (Workday) → employees[]
  • GL chart of accounts → accounts[]
  • Cost-center hierarchy → bizLines[], markets[]

Outbound

  • Oracle EPM, Anaplan, Workday Adaptive (P&L by chartfield × month)
  • BI dashboards (real-time aggregations)
  • Data lake (full state snapshots)
  • XLSX / PDF export

Connector pattern (target)

Every integration is a versioned connector module shipping with:

  • Schema map (Trestle field → target field)
  • Chartfield crosswalk
  • Sync mode: incremental + full-load
  • Retry / backoff with idempotency keys
  • Reconciliation report (counts, sums, deltas)

Today vs target

  • Today: ad-hoc per integration; not versioned.
  • Target: connector registry; new firm = configure instance, not fork.

Opens

  • Vendor schema-change handling and version pinning.
  • SLA per connector (freshness, retry budget).
  • Vendor sandbox testing in CI.

6.5 Vendor / T&E / Other (consolidated)

Shared shape

rows[] with type discriminator (vendor type · expense type · C&B vs OAO), description, chartfields, jan-dec.

Per-module differences

  • Vendor / OAO: parent co + vendor lookup against admin catalog.
  • T&E: expense-type list (admin reference).
  • Other: split by C&B vs OAO at the type level.

Calc

Direct monthly entry · aggregate to module total → P&L line.

Reference data

Vendor catalog · expense-type list · GL accounts · type taxonomy.

Today vs target

  • Today: three separate row arrays in state_data; logic copy-pasted.
  • Target: one shared expense_rows table with type + sub_type columns · one shared grid component · each module is a configured view.

Edge cases & opens

  • PO splits across chartfields.
  • Multi-currency invoices.
  • Mid-year vendor name / consolidation changes.
  • Open: should Contractors fold into the same shape? Currently no — has rate × hours and CapEx split. Worth re-examining once the shared table exists.
