Trestle — System Brief & Submodule Inventory

Shared reference for technical and finance / finance-tech leads supporting the rebuild. Each submodule carries a complexity gauge and a "similar to" tag list so reviewers can scan where depth matters.

1. Exec Summary

Trestle is a multi-user FP&A planning platform: one shared workspace per fiscal year holding 14 plan files (1 AOP + 12 monthly Rolling Forecasts + 1 Long-Term Plan). Each plan file is a state_data JSON containing every input module (revenue, headcount, vendor, contractor, T&E, other, D&A) coded to a shared chartfield grid (Business Unit / Biz Line / Market / Project / Account). Calc outputs feed a P&L walk and a pivot/analytics layer; collaboration runs over WebSocket presence with live cursors and shared state.

The rebuild target preserves the conceptual model but replaces the single-blob state with a normalized data spine, layered access control, and standard EPM connectors. This brief is the shared map: an inventory plus deep dives on the highest-leverage submodules.

2. System Spine

Plan Container

Workspace (year) holds 14 plan files: 1 AOP · 12 monthly RFs · 1 LTP. Each plan = one state_data JSON. Scenario types: budget forecast actual ltp.

Input Modules

Revenue · HC & Comp · Vendor / OAO · Contractors · T&E · Other · D&A. Each row coded to chartfields (BU, Biz Line, Market, Project, Account) plus month columns Jan-Dec.

Outputs & Collab

P&L walk · pivot / analytics · scenario compare. Real-time collab via WebSocket presence, live cursors, shared state. Admin and access controls layered on top.

5. Deep Dives

First-pass depth on the five priority submodules. Intentionally shallow — we iterate from here.

5.1 HC & Compensation

Business framing

The most-edited and most-material module. Drives compensation, benefits load, CapEx capitalization, and FTE rollups. Used by HR-aligned planners alongside finance.

Data shape

employees[] — name, salary, seniority, function, country, capPct, allocations[], start mo, end mo, bonus %, equity %.

Calc rules

  • Monthly proration based on start / end month (mid-month handled as full month today; flag for revisit).
  • Bonus % applied as annual amount and posted to a target bonus month (configurable).
  • Benefits load applied as country-specific % on top of base salary.
  • CapEx % (capPct) splits cost between CapEx and OpEx; flows to D&A pipeline.
  • Allocation % across functional pillars routes cost lines into pivot dimensions.

Dependencies

Salary bands (admin reference) · benefits load by country (admin reference) · FX rates (future, not in v1) · pillar map · GL accounts.

Today vs target

  • Today: nested in state_data JSON; full-plan blob save on every edit.
  • Target: dedicated employees table with row-level versioning · reference_sets for bands / benefits / FX · per-employee audit trail.

Edge cases & open questions

  • Mid-month start / end and partial-month proration policy.
  • BU / market transfers mid-year (does the row split, or one row with allocation change?).
  • Equity refresh accounting and overlay vs base.
  • Contractor → employee conversion and continuity.
  • Multi-currency comp and FX restatement.

5.2 Collaboration & Presence

Business framing

Real-time multi-user editing during close and forecast cycles. Several finance users editing the same plan concurrently; loss of edits or stale views breaks trust in the data.

Today's shape

  • WebSocket presence + live cursors.
  • Full state_data JSON saved on change; last-write-wins.
  • No cell-level merge; no row-level locking.

Problems with today

  • Lost edits under concurrent saves.
  • Large WS payloads at scale.
  • Audit trail only in client state, not in DB.

Target architecture

  • Per-cell CRDT (or OT) doc per plan.
  • Redis pub / sub for fan-out across app instances.
  • Optimistic locking with version column; server-authoritative merge.
  • Append-only audit log partitioned by month (user, field, before, after, ts).

Flow

Client emits ops → server validates against tenant + RLS → applies to CRDT doc → broadcasts via Redis → other clients merge → batched persist.

Risks & opens

  • Replay correctness across schema evolution mid-edit.
  • Presence performance at hundreds of users.
  • Audit log growth and partitioning policy.
  • Open: which CRDT lib · cell vs row granularity · how to surface "X is editing Y" cleanly.

5.3 P&L Walk

Business framing

The canonical financial output. Every input module flows through here. Correctness here is the headline measure of system trust.

Inputs

Every module's calc output: Revenue, C&B (from HC), OAO (from Vendor + Other), CTR (from Contractors), T&E, Other C&B, D&A.

Output

Chartfield × month grid with subtotal hierarchy (gross margin, opex, EBITDA, etc.) and scenario variance overlay.

Logic

  • Sum by chartfield × month per line.
  • Subtotal hierarchy defined per tenant template.
  • Loaded-cost rollups (e.g., bonus + benefits into C&B).
  • Scenario compare (budget vs forecast vs actual) with variance overlay.

Today vs target

  • Today: computed in client memory from state_data.
  • Target: server-side aggregation against normalized tables · materialized views per scenario · cached at chartfield rollup level.

Risks & opens

  • Any module shape change ripples here — needs contract tests against golden P&L fixtures.
  • Open: report builder vs hard-coded P&L lines · tenant-specific subtotal hierarchies · drill-through UX.

5.4 EPM & Integrations

Business framing

Keep Trestle in sync with the firm's systems of record (ERP, HR, EPM, BI). New firms should onboard by configuring connector instances, not writing new code.

Inbound

  • ERP actuals (SAP / Oracle / NetSuite) → posted to state.actuals.
  • HR (Workday) → updates employees[].
  • GL chart of accounts → accounts[].
  • Cost-center hierarchy → bizLines[], markets[].

Outbound

  • Oracle EPM, Anaplan, Workday Adaptive (P&L by chartfield × month).
  • BI dashboards (real-time aggregations).
  • Data lake (full state snapshots).
  • XLSX / PDF export.

Connector pattern (target)

Every integration is a versioned connector module shipping with:

  • Schema map (Trestle field → target field).
  • Chartfield crosswalk.
  • Sync mode: incremental + full-load.
  • Retry / backoff with idempotency keys.
  • Reconciliation report (counts, sums, deltas).

Today vs target

  • Today: ad-hoc per integration; not versioned.
  • Target: connector registry; new firm = configure instance, not fork.

Opens

  • Vendor schema-change handling and version pinning.
  • SLA per connector (freshness, retry budget).
  • Vendor sandbox testing in CI.

5.5 Vendor / T&E / Other (consolidated)

Business framing

Three modules sharing one row shape — line item + chartfields + monthly amounts. Documented once because the rebuild produces one shared component used three ways.

Shared shape

rows[] with type discriminator (vendor type · expense type · C&B vs OAO), description, chartfields, jan-dec.

Per-module differences

  • Vendor / OAO: parent co + vendor lookup against admin catalog.
  • T&E: expense-type list (admin reference).
  • Other: split by C&B vs OAO at the type level.

Calc

Direct monthly entry · aggregate to module total → P&L line.

Reference data

Vendor catalog · expense-type list · GL accounts · type taxonomy.

Today vs target

  • Today: three separate row arrays in state_data; logic copy-pasted.
  • Target: one shared expense_rows table with type + sub_type columns · one shared grid component · each module is a configured view.

Edge cases & opens

  • PO splits across chartfields.
  • Multi-currency invoices.
  • Mid-year vendor name / consolidation changes.
  • Open: should Contractors fold into the same shape? Currently no — has rate × hours and CapEx split. Worth re-examining once the shared table exists.
